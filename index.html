The Thesis

Classical or Quantum.

System size dictates governance. Not the same rules at higher volumes. Different rules entirely.

Personal AI · Classical

Model the whole system.

  • You can enumerate every job.
  • You can predict any single call.
  • Guards are deterministic hooks.
  • State is traceable end to end.
  • Failures are one bot, one fix.

Enterprise AI · Quantum

Measure the distribution.

  • You cannot enumerate every call.
  • You cannot predict any one of them.
  • Guards are sampled review.
  • State is aggregate metrics.
  • Failures are drift across a population.

Different worlds, different toolkits, all AI.

The thesis. Personal AI is small enough that you can model the whole system. You know every job. You can predict any single call before it runs. You guard with hooks, deterministic gates that pass or fail. Classical mechanics. Enterprise AI at scale is the opposite. You cannot predict any single call. You can only measure distributions. You watch for drift. You sample for review. You build calibration sets. Quantum mechanics. The mistake I see people make. They treat enterprise compliance as personal compliance with bigger budgets. It is not. Different physics. The toolkit is different. The rules are different. The success criteria are different. That is the wall I want to walk you across tonight. Part one is the classical side. Part two is the quantum side. The bridge between them is one principle that holds on both sides.

Personal AI · Reasoning Loop

The Algorithm.

Seven phases. Intent echoing. ISA scaffold. Verification before claim.

1
OBSERVE Echo intent. Preflight gates. Reproduce first.
2
THINK Premortem. Riskiest assumptions surface.
3
PLAN Scope. Deliverables. Parallelism. Feature breakdown.
4
BUILD Invoke capabilities. Root cause at ingestion.
5
EXECUTE Per-criterion inline verification. Tool probes per transition.
6
VERIFY Live probe artifacts. Re-read user. Conflict surfacing.
7
LEARN Conjecture, refutation, learning. Changelog. Doc sync.

The agent is required to follow this loop on any non-trivial task. The loop is auditable.

The reasoning loop the agent has to run on any non-trivial task. I call it the JAX Algorithm. Version six point three point zero. Seven phases. OBSERVE is intent echoing. Restate what the user asked in one sentence. Preflight gates fire here. Reproduce the bug before you touch code. THINK is premortem. Surface the riskiest assumptions. Knowledge check. PLAN is scope, deliverables, parallel opportunities, feature breakdown. BUILD is invoke the capabilities the plan called out. EXECUTE is per-criterion inline verification. Every named criterion gets a tool probe at transition. VERIFY is live probe of user-facing artifacts. Re-read the original message. Surface conflicts. LEARN is conjecture, refutation, learning. Changelog gets an entry. Why is this compliance? Because the loop is auditable. Every phase has a structured artifact. When something goes sideways, I can trace back to which phase the agent skipped or got wrong. The loop is the difference between an agent that improvises and an agent that runs a process you can review.
Personal AI · Risk Classifier

Action Trust.

Every shell command classified before it runs. Three risk levels. 26 patterns.

LOW
Silent passthrough. No output. Read-only, local, harmless.
MEDIUM
Allow with informational warning. Reversible state changes. API calls. curl, git commit, bun install.
HIGH
Strong confirmation context injected. Hard to reverse or shared production state. git push, rm -rf, sudo, SQL DROP.
// Pre-tool-use hook. Action Trust Model.
cmd: git push origin main --force
classify: HIGH
! confirm: this rewrites remote history
! check: force-push to main, review branch protection

cmd: curl http://localhost:3000/api/health
classify: LOW // local exemption

// Trust never blocks. It always allows.
// It injects context the agent must read.

Never blocks. Forces visibility through context, not control.

Action Trust. A pre-tool-use hook that classifies every shell command before the agent runs it. Three levels. Twenty six patterns. LOW is silent. Read-only. Local. Harmless. The agent runs it, no friction. MEDIUM is allow with a warning. Reversible state changes. API calls. Git commits. Package installs. The warning shows up as additional context. The agent reads it before acting. HIGH is allow with strong confirmation context. Hard to reverse. Shared or production state. Git push. SQL drop. Recursive deletes. Sudo. The warning is loud. The agent has to acknowledge it in its reasoning before running. Key design choice. Action Trust never blocks. It always allows. It injects context. The agent could ignore the context. It does not, because the context is in its working memory and it has to argue around it to skip. That is enough governance for ninety percent of cases. The remaining ten percent are handled by ContainmentGuard on the next slide, which does block. The local exemption is real. Curls to localhost get downgraded to LOW. Otherwise every dev loop becomes a warning storm.

Mode progression. Every autonomous bot I ship has this dial. Five modes. Off means killed. The cron still fires, the wrapper still runs, the script exits before doing work. Dry means generate drafts but post nothing. Logs only. Review means render drafts and post them to a Slack channel for me to approve one at a time. Test means post to a sandbox community that has no real members. Live means production. One environment variable controls the entire behavior. The kill switch is one string flip. If a bot starts misbehaving at three in the morning, one Cronicle setting change and it is off. No code change. No deploy. No login to a vendor. Every bot earns its way up. Off when I write it. Dry while I tune the prompt. Review for at least a week with me as the human approver. Test in a sandbox. Then live. And even live ships with a paranoia cap. One post per run for the first twenty four hours. This pattern came from a real incident. A bot that worked fine in test broke at live scale in a subtle way, on a real community, in front of real strangers. Now every bot walks the ladder.

ContainmentGuard.

Six zones. Pre-tool-use hook. Exit code 2 means denied.

Identity and credentials cannot leak past these zones.

Z1user-dataPAI/USER paths
Z2config-secrets.env, settings.json, .mcp.json
Z3runtime-memorybrain, memory, projects
Z4private-skills_underscore-prefixed

The hook runs on every file write. The agent cannot bypass it.

ContainmentGuard. The one hook that does block. Pre-tool-use hook on Edit, Write, and MultiEdit. It scans the content the agent is about to write to disk. If the content matches an identity string or a credential value prefix, exit code two. The write does not happen. Six zones define what is sensitive. User data. Config secrets. Runtime memory. Private skills. Install state. Private infrastructure. Anything outside those zones must stay clean. So if the agent tries to copy a snippet of my email or an API key prefix into the public github folder, the hook blocks it. Patterns are deterministic. My real email. The literal prefix patterns of every credential type I use. Supabase service role keys start with eyJ. Resend keys start with re underscore. Anthropic keys start with sk-ant. Cloudflare tokens have a known shape. These prefixes are easy to scan. Tuning trick worth stealing. I shipped the hook in log-only mode for the first twenty four hours. It logged what it would have blocked but did not block. That let me tune the pattern list against real traffic before I flipped enforcement on. First-day paranoia cap pattern. A policy that says "do not paste secrets" gets ignored on a tired Tuesday. A hook that physically cannot write the secret is on duty forever.
Personal AI · Compliance as Filters

The Filter Chain.

Welcome bot decisions encoded as composable, independently togglable filters.

1Max age48 hours
2Min age5 minutes
3Max per run3 intros
4Skip if commented≥ 20 comments
5Min body length20 chars
6Topic classifieron/off per community
7Coin flip85% post rate

Why this is compliance:

1

Each filter is testable in isolation. No "the bot did a thing." It was filter 4 that decided to skip.

2

Every filter is independently togglable. Per community. Per environment. Per audit ask.

3

The chain is versioned in code. Every decision goes to Supabase with the filter version that processed it.

The filter chain. Welcome bot that runs across seven Skool communities. Every thirty minutes it fetches the newest intros, walks them through seven filters, and decides whether to post a welcome comment. The seven. Max age forty eight hours. Do not reply to two-week-old zombie threads. Min age five minutes. Do not be the first comment within seconds, that reads as a bot. Max per run three intros. Stops sudden floods. Skip if twenty or more comments. The party already started, no value adding. Min body twenty characters. Skips the wave-emoji "hi" posts. Topic classifier, on or off per community based on volume. And finally an eighty five percent coin flip. A bot that comments on literally every intro looks mechanical. Why this is compliance. Three reasons. One. Each filter is testable in isolation. When the bot does not post on something I expected, I can run the chain in test mode and it tells me exactly which filter dropped the decision. There is no "the bot did a weird thing." It is filter four said twenty three comments already. Two. Each filter is independently togglable. Different communities have different rules. One has a ninety six hour max age. Another runs the classifier off because intros are noisy. The dial is per community. Three. The chain is in version control. Every filter change is a commit. Every decision goes to a Supabase table with the chain version that processed it. If someone asks what happened on May tenth at six AM, I can answer.

Voice rules and a review gate. Two layers. The compliance problem. If a bot is writing as me, how do I make sure it does not fabricate things about me? Real example. The welcome bot drafted a comment that included the phrase "just starting to layer in Claude Code." That is false. I use Claude Code all day. The bot was extrapolating from the member's intro and mirroring their journey back to sound relatable. The catch happened in review mode. In live mode, it would have shipped. Layer one is voice rules enforced in code. A function called violatesRules checks every draft. Three accepted patterns. Pattern A is a welcome paragraph capped at forty words. Pattern B is greeting plus two or three short paragraphs. Pattern C is a direct question or a brief list. Plus a word blacklist and a ban on gerund openers, the -ing words that read as AI slop. Drafts that fail any rule get marked not okay. Layer two is manual review. Mode flips from dry to review. Drafts get posted to a Slack channel and wait for me to approve. The fabrication was caught here. Not by a clever allowlist. By me reading the draft. The honest design. There is no automatic provenance allowlist for biographical claims yet. The rules catch the easy stuff. The Slack gate catches the rest. The next layer would be exactly that, a provenance allowlist for first-person claims. Roadmap item.

External communications gate. Same posture as Action Trust. Inject context, do not block. Pre-Bash hook. Nine trigger patterns. Resend, GoHighLevel, Slack API, curl to email APIs, Twilio, WhatsApp, Gmail CLI, SMS sending. If the command matches, the hook injects a system reminder that says: show me the recipient, the subject, the full body, and wait for explicit approval before running. Four safe channels are exempt. Internal alert channels like jax-alerts and jax-daily. The gate would warn on every cron heartbeat otherwise. This is a deliberate design choice. I could hard block external sends, but then the agent loses the ability to draft, queue, and prep. The forced visibility pattern lets it prep everything, show me the artifact, and wait. I read, I say send it, it sends. Why is this compliance? Same reason an exec wants to see a press release before it ships. The cost of one bad outbound message is real money or real relationships. The cost of one extra read is fifteen seconds.

Memory As Audit Trail.

You cannot govern what you cannot query later.

1
CAPTURE Hooks emit on every tool use, agent spawn, error, cost event.
2
PERSIST SQLite FTS5 + nomic embeddings. Sub-agent stop hook persists findings.
3
DECAY Episodic 2%/day. Semantic 0.5%/day. Pruned below 0.1.
4
QUERY Searchable, citable. "What did the bot decide on May 10?"

Time-based rejections marked transient, never written to the seen list. Audit reflects reality.

Memory as audit trail. The receipts. Pipeline. Hooks emit every event to a bus. Sessions, tool uses, agent spawns, errors, costs. The sub-agent stop hook extracts findings from agent transcripts and writes them as structured rows. SQLite database with FTS5 full-text search and nomic embeddings holds it all. Decay runs nightly. Episodic memories fade two percent per day, semantic ones fade slower, and anything that drops below a salience floor of point one gets pruned. The point. I can answer real questions about my own AI's behavior. Show me everything the welcome bot did last week. Show me every external email the AI tried to send. Trace this Skool comment from filter decision to Supabase log row. One subtle thing worth holding. Rejections must distinguish transient from permanent. If a time-based filter says this post is too young to comment on, that is transient. Mark it not-yet, do not mark it seen. Otherwise the post becomes invisible forever when it matures. The audit log has to reflect reality. Same shape as enterprise audit logs. Mine is a SQLite file. Theirs is a Splunk index. Same engineering discipline. Different cost.
Bridge · Load Bearing

Reduce AI As Much As Possible.

Every regex you write is one prompt you do not. Every SQL you write is one drift surface you avoid.

✅

Default to deterministic code.

Clean the data. Write the script. Modularize the pieces that build on each other. The same input produces the same output.

💣

AI is the exception.

Every place a regex, a SQL query, or a Bun script can replace an LLM call, replace it. The LLM is for the ambiguity, not the math.

This rule holds on both sides of tonight's wall.
Personal or enterprise. Classical or quantum.

The bridge. The one principle that holds on both sides of tonight's wall. Reduce AI as much as possible. Default to deterministic code. On the personal side, every filter in that seven-step chain is deterministic. Age comparisons. Character counts. Comment counts. The LLM only enters when the work is genuinely ambiguous. Topic classification. Voice drafting. And even then, the output gets checked by deterministic rules before it ships. On the enterprise side, the same rule applies. Every place a SQL query can answer a question, that is where the answer should come from. Every place a regex can extract a field, that is where it should be extracted. The LLM is for the parts you genuinely cannot model deterministically. The judgment, the ambiguity, the language work. Why this matters for compliance. Deterministic code is the only thing that is auditable in the strong sense. Same input, same output, traceable. LLM calls are probabilistic. Auditing them requires distributions, sampling, and drift detection, which is what part two is about. So the design rule. Push as much logic as you can to deterministic code first. Use the LLM only where it is the right tool. Then audit the LLM portion with the quantum toolkit we are about to walk through.

The personal stack assumes you can predict any single call. At scale, you cannot.

100 calls per day

Eyeball the audit log. Replay any failure. Hooks per action. One bot, one fix.

100,000 calls per day

You cannot read every log. You can only watch distributions and sample.

The wall is not stakes. The wall is physics.

The crossing. Why the toolkit has to change. On the personal side, I can eyeball the audit log. I can replay any failure. I can put a hook on every tool use. One bot, one fix. The system is small enough that I can model the whole thing in my head. At enterprise scale, that is impossible. You cannot read the log of every call. You cannot predict any single call before it runs. You can only watch distributions and sample for review. The patterns that worked on the personal side do not transfer. Concrete example. The layered kill switches work for one bot I personally babysit. They do not work for a thousand bots run by a hundred operators. The unit of governance has to move from the bot to the population. That is the wall. Not "same patterns, higher stakes." Different patterns entirely. Personal AI is classical. Enterprise AI is quantum. Different physics. Different toolkit. You cannot predict any single call. That is the entry point for everything in part two. Three shifts. One. You measure distributions, not events. Hit rates. Confidence histograms. Per-rule fire counts. Aggregate stats, not anecdotes. Single bad calls happen. The question is the rate. Two. Drift is the dominant bug class, not crashes. Same prompt, same input, two weeks later you get a different output. The model moved underneath you. Or the data did. Or both. Drift is silent. Crashes are loud. Drift wins. Three. You sample, you do not audit. Random small-N blind human review on a continuous schedule. Five samples a day, blind, scored. Catches the qualitative regressions that dashboards never surface. Enterprise · Drift

Drift Detection = Replay The Golden Set.

Vendor status pages will not tell you the model regressed. The eval set will.

// Same golden set. Same prompt. Two weeks apart.
[2026-04-15] v4 prompt vs sonnet-4-6: 76.2% agree
[2026-04-29] v4 prompt vs sonnet-4-6: 71.4% agree

! delta: -4.8 points in two weeks
! action: investigate sample-level deltas
! action: schedule prompt patch + re-eval

// Vendor never told us. The replay did.

Why replays beat status pages.

  • • Vendors do not announce small regressions or A/B holdouts.
  • • Your prompt-model pair is what regressed, not the model alone.
  • • Per-sample deltas tell you what changed, not just that something did.
  • • The replay infrastructure doubles as your vendor swap toolkit.
Drift detection. The toolkit is the same replay infrastructure as the Prompt Lab. Method. Same golden set. Same prompt. Run on a schedule against the current vendor model. Compare against the last run. If the agreement rate drops meaningfully, you have drift. Why this matters. Vendor status pages will not tell you a model regressed on your particular task. They will tell you about outages. They will not tell you that a quiet model update degraded your accuracy by five points. Your prompt and model pair regressed, not the model in the abstract. The only thing that catches this is replaying your own set. Per-sample deltas tell you what changed, not just that something did. Same one hundred eighty six sample subset. Five of those changed from agree to disagree this run. Look at those five. That is your investigation surface, not the aggregate. Bonus. The same infrastructure is your vendor swap toolkit. The day you want to move from Anthropic to OpenAI to Google to whoever, you run all three against the golden set and compare. The decision is data-driven, not vibes. This is the difference between knowing your AI works today and knowing it still works next month.
Enterprise · Sampled Review

Blind Human Sampling, Continuously.

Dashboards lie by aggregation. Random blind review catches what they hide.

The mechanic.

  • • Random small-N draw from yesterday's decisions.
  • • Same blind UI as the SME calibration tool.
  • • Human scores it. AI decision revealed second.
  • • Disagreements feed the prompt patch queue.

The same infrastructure that bootstrapped the golden set now keeps it honest.

Why dashboards alone fail.

1

Aggregate accuracy stays flat while a category quietly collapses.

2

Confidence scores are vibes, not calibration, until a human disagrees and you measure.

3

Tone, format, and framing drift never show up in metrics. Only in reads.

Blind human sampling. Continuously. Mechanic. Every day, random small-N draw from yesterday's decisions. Same blind UI as the SME calibration tool. Human scores first. AI decision gets revealed second with the human's commentary captured. Disagreements feed the prompt patch queue. The cost is low. Five to ten samples a day takes a reviewer fifteen minutes. The catch rate is high because the dimensions you are measuring are exactly the ones that aggregate dashboards miss. Why dashboards alone fail. Three reasons. One. Aggregate accuracy stays flat while a category collapses. You can lose ten points on a low-volume category and never see it in the headline number. Two. Confidence scores are vibes until a human disagrees. The model says ninety percent confident. The human says wrong. Now you know what ninety percent actually means. Without that disagreement, it is just a number. Three. Tone, format, and framing drift never show up in metrics. Only in reads. The AI starts hedging too much, or losing structure, or sounding less like the brand. Nobody's eval set scores tone. A human reading the sample catches it on day one. Notice the leverage. The same blind UI we built for SME calibration now runs blind review every day. The infrastructure compounds.
